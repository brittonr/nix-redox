## Why

Every profile hardcodes service startup logic in `init-scripts.nix` with scattered `lib.optionalAttrs` blocks. Adding a service means editing the build module, not the profile. The existing `services.services` structured type was added but no profile uses it — all 18 profiles set raw `initScripts` or rely on the hardcoded conditionals. Profiles should declare *what* services they want, and the build system should figure out the init scripts.

## What Changes

- **Migrate hardcoded services to declarations**: The networking daemons (smolnetd, dhcpd), console (getty), graphics (orbital, audiod), scheme daemons (stored, profiled), and pcid-spawner are currently generated by conditional logic in `init-scripts.nix`. Convert each to a structured `services.services` entry declared by the relevant module (networking.nix, graphics.nix, snix.nix, etc.), so enabling `networking.enable = true` automatically declares the smolnetd service.
- **Add dependency ordering**: Services need `after` dependencies (e.g., orbital after inputd, getty after ptyd). The current numbered prefix (`00_base`, `10_net`, `20_orbital`) is implicit ordering — make it explicit with `after` fields, and let the build system compute the numbering.
- **Add environment variables per service**: Some services need specific env vars (e.g., `VT=3` for orbital, `XDG_CONFIG_HOME=/etc` for getty). Currently handled by `export` lines in raw init scripts. Add an `environment` attrset to the service type.
- **Track services in manifest with full metadata**: The manifest currently stores only init script names (`services.initScripts`). Store the full service declarations so `snix system switch` can diff service-level changes (not just script-level).
- **Service change reporting in activation**: When services are added/removed/changed between generations, the activation plan should show the service name and type, not just the init script filename.

## Capabilities

### New Capabilities
- `declarative-services`: Module-level service declarations with dependency ordering, environment injection, and manifest tracking. Replaces hardcoded init script generation with composable per-module service definitions.

### Modified Capabilities
- `guest-rebuild`: Activation plan displays richer service diffs (name, type, description changes vs. raw script name diffs).

## Impact

- **Files**: `nix/redox-system/modules/services.nix` (expanded service type), `nix/redox-system/modules/build/init-scripts.nix` (rewritten to consume declarations), `nix/redox-system/modules/networking.nix`, `nix/redox-system/modules/graphics.nix`, `nix/redox-system/modules/snix.nix` (declare their services), `nix/redox-system/modules/build/manifest.nix` (richer service metadata), `snix-redox/src/activate.rs` (service diff display), `snix-redox/src/system.rs` (manifest struct update).
- **APIs**: `serviceType` gains `after`, `environment` fields. Manifest JSON `services` section gains full service objects instead of bare names.
- **Dependencies**: None added.
- **Testing**: Existing functional-test and all test profiles must produce identical init scripts after migration (regression gate). Build checks validate service dependency graph is acyclic.
