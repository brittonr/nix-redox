## ADDED Requirements

### Requirement: Host-side HTTP binary cache server for testing

The test infrastructure SHALL start a Python HTTP server on the host that serves a binary cache directory. The server SHALL bind to `0.0.0.0` on a known port (default 8080) so the QEMU guest can reach it at `10.0.2.2:8080` via SLiRP networking. The binary cache SHALL be generated by the existing `build-binary-cache.py` infrastructure with at least 3 test packages.

#### Scenario: HTTP server starts before VM boot
- **WHEN** the network install test runs
- **THEN** the host SHALL start `python3 -m http.server 8080` serving the binary cache BEFORE the VM boots

#### Scenario: Guest can reach host HTTP server
- **WHEN** the guest has completed DHCP and has network connectivity
- **THEN** the guest SHALL be able to HTTP GET `http://10.0.2.2:8080/packages.json`

#### Scenario: Server serves narinfo files
- **WHEN** the guest requests `http://10.0.2.2:8080/{hash}.narinfo`
- **THEN** the server SHALL respond with the narinfo file content

#### Scenario: Server serves NAR files
- **WHEN** the guest requests `http://10.0.2.2:8080/{hash}.nar.zst`
- **THEN** the server SHALL respond with the compressed NAR file

### Requirement: QEMU SLiRP networking for test VMs

The network install test SHALL use QEMU user-mode networking (SLiRP) to provide the guest with network access. The QEMU invocation SHALL include `-netdev user,id=net0 -device e1000,netdev=net0`. No TAP interface or root access SHALL be required.

#### Scenario: Guest gets IP via DHCP
- **WHEN** the VM boots with SLiRP networking
- **THEN** the guest SHALL obtain IP 10.0.2.15/24 via DHCP from smolnetd

#### Scenario: Guest DNS works
- **WHEN** the guest resolves a hostname
- **THEN** DNS queries SHALL be forwarded through the SLiRP gateway at 10.0.2.3

### Requirement: Network install test profile

A test profile SHALL exist that includes networking drivers (e1000d), the network stack (smolnetd, dhcpd, dnsd), and a test script that exercises remote binary cache operations. The profile SHALL NOT include the packages that will be installed via the network (they must come from the remote cache, not be baked into the image).

#### Scenario: Test profile boots with networking
- **WHEN** the network test profile boots in QEMU with SLiRP
- **THEN** the guest SHALL have a working network stack with DHCP-assigned IP

#### Scenario: Test packages not pre-installed
- **WHEN** the test profile boots
- **THEN** the packages to be installed via remote cache (e.g., ripgrep, fd) SHALL NOT be present in the guest image

### Requirement: In-guest network install verification tests

The test script SHALL run inside the Redox VM and verify end-to-end network package installation. Test results SHALL be emitted as `FUNC_TEST:<name>:PASS` or `FUNC_TEST:<name>:FAIL` on the serial console, matching the existing functional test output format.

#### Scenario: Network connectivity test
- **WHEN** the test script starts
- **THEN** it SHALL verify network connectivity by checking DHCP completion and HTTP reachability of the cache server

#### Scenario: Remote search test
- **WHEN** the test script runs `snix search --cache-url http://10.0.2.2:8080`
- **THEN** it SHALL verify that available packages are listed

#### Scenario: Remote install test
- **WHEN** the test script runs `snix install <package> --cache-url http://10.0.2.2:8080`
- **THEN** it SHALL verify the package binary exists in the profile and executes correctly

#### Scenario: Remote install creates store path
- **WHEN** a package is installed from the remote cache
- **THEN** the store path SHALL exist under `/nix/store/` and be registered in PathInfoDb

#### Scenario: Remote install idempotency
- **WHEN** `snix install <package>` is run twice for the same package
- **THEN** the second invocation SHALL report "already installed" without re-downloading

### Requirement: Test result collection from serial console

The test infrastructure SHALL use the same serial console polling approach as the existing functional tests: run the VM with `--serial file=<path>`, poll the log file with grep for `FUNC_TEST:` markers, and report pass/fail counts.

#### Scenario: All tests collected
- **WHEN** the VM completes the test script and shuts down
- **THEN** the host SHALL parse all `FUNC_TEST:` lines and report the summary

#### Scenario: Test timeout
- **WHEN** the VM does not produce all expected test results within the timeout
- **THEN** the test SHALL fail with a timeout error showing which tests were not completed
